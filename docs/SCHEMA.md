# Database schema

SQLite, migration `001`, defined in
[`src-tauri/src/db/schema.sql`](../src-tauri/src/db/schema.sql).

## Conventions

- **Every timestamp is `INTEGER` epoch milliseconds, UTC.** No `TEXT` dates, no
  local time, anywhere.
- **Ids are `TEXT`**, ULID-shaped: a 48-bit millisecond timestamp plus 80 bits
  of entropy in Crockford base32. They sort chronologically, so indexes give
  time ordering for free.
- **Foreign keys cascade.** Deleting a session cannot orphan a row.
- **Booleans are `INTEGER` with a `CHECK (x IN (0,1))`.**

## Pragmas

```sql
journal_mode = WAL        -- readers never block the writer
synchronous  = NORMAL     -- crash-safe under WAL
foreign_keys = ON
temp_store   = MEMORY
cache_size   = -65536     -- 64 MiB
mmap_size    = 268435456  -- 256 MiB
busy_timeout = 5000
```

## Tables

### `sessions`

The central record. One imported session becomes exactly one row.

| Column | Type | Notes |
| --- | --- | --- |
| `id` | TEXT PK | ULID-shaped |
| `title` | TEXT | |
| `project` | TEXT | clustering key for the galaxy |
| `folder` | TEXT? | absolute path when known |
| `summary` | TEXT | generated by importers, user-editable |
| `notes` | TEXT | **never overwritten by re-import** |
| `language` | TEXT? | canonical id from the language registry |
| `status` | TEXT | `active` \| `completed` \| `paused` \| `archived` |
| `kind` | TEXT | `star` \| `planet` \| `asteroid` |
| `complexity` | REAL | 0..1, drives node radius |
| `importance` | REAL | 0..1, drives node brightness |
| `pinned` | INTEGER | |
| `source` | TEXT | importer id, e.g. `core:claude-jsonl` |
| `source_ref` | TEXT? | original path |
| `started_at` | INTEGER | when the *work* happened |
| `ended_at` | INTEGER? | |
| `updated_at` | INTEGER | |
| `imported_at` | INTEGER | |
| `message_count` | INTEGER | denormalised |
| `file_count` | INTEGER | denormalised |
| `artifact_count` | INTEGER | denormalised |
| `token_estimate` | INTEGER | |
| `content_hash` | TEXT **UNIQUE** | the dedupe key |

**Why the denormalised counters.** The galaxy renders from a single indexed
scan of this table with zero joins. At 100k sessions that is the difference
between an instant load and a multi-second one. They are recomputed from the
child tables on every write, so they cannot drift.

**Why `content_hash` is UNIQUE.** Re-importing the same export is a constraint
violation, which the bulk importer catches and reports as "already in library"
rather than creating a duplicate star. The hash is derived from content only —
never from ids or import time — so it is stable across machines.

Indexes:

```sql
idx_sessions_started    (started_at DESC)
idx_sessions_project    (project, started_at DESC)
idx_sessions_language   (language)          WHERE language IS NOT NULL
idx_sessions_status     (status)
idx_sessions_pinned     (pinned)            WHERE pinned = 1
idx_sessions_updated    (updated_at DESC)
```

The two partial indexes stay small because most rows are neither pinned nor
have a null language.

### `artifacts`

Non-conversational output. One table rather than six because they share a shape
and it keeps "what did I decide?" to a single indexed scan.

`kind ∈ {code, architecture, decision, todo, command, note}`, plus `title`,
`language`, `content`, `path`, `message_seq` (for jump-to-context) and `done`
(meaningful only for todos).

### `files`

Files the session touched. `UNIQUE (session_id, path)`; inserts upsert, keeping
the more informative action (`created` beats `referenced`) and the longer
snippet.

`idx_files_path` exists because "which sessions touched this file?" is a
first-class question, asked across the whole library rather than within a session.

### `tags`, `session_tags`, `technologies`, `session_technologies`

Standard many-to-many. Names are `COLLATE NOCASE UNIQUE`, so `React` and
`react` collapse to one row. Ids are `t_`/`k_` plus a truncated SHA-256 of the
lowercased name, which makes them deterministic across machines — an exported
and re-imported library reuses the same tag rows.

### `links`

Knowledge-graph edges: `(from_id, to_id, kind, weight)` with
`kind ∈ {same-project, shared-files, shared-tech, continuation, manual}`.

Derived edges are recomputed wholesale by `replace_links`, which deletes
everything **except** `kind = 'manual'` — those are user data.

### `kv`

`(key, value, updated_at)`. Holds settings, plugin configuration and saved
views as JSON. One row for the whole settings document keeps reads to a single
lookup and makes the configuration trivially exportable.

## Full-text search

```sql
CREATE VIRTUAL TABLE session_fts USING fts5(
    title, project, summary, notes, tags, technologies, body,
    session_id UNINDEXED,
    tokenize = 'unicode61 remove_diacritics 2'
);

CREATE TABLE fts_map (session_id TEXT PRIMARY KEY, rowid_ref INTEGER NOT NULL);
```

**Column order is load-bearing.** It must match `SEARCH_FIELDS` in
`src/search/document.ts`, because `bm25()` weights are positional:

```sql
bm25(session_fts, 10.0, 6.0, 4.0, 4.0, 5.0, 3.0, 1.0)
--                title project summary notes tags tech body
```

Body is weighted lowest by a wide margin: transcripts are long, and without
damping every session matches everything.

**Why the index is maintained in application code, not triggers.** The indexed
`body` is not any single column — it aggregates artifacts, file paths and a
size-capped sample of messages, assembled by `buildDocument()` in TypeScript.
Keeping that in one place means the SQLite index and the in-memory index can
never disagree about what was indexed.

**Why `fts_map`.** FTS5 has no `UPSERT`, so updating a row means delete-then-
insert by `rowid`. `fts_map` stores that rowid per session, avoiding a scan of
the virtual table on every edit.

**Body cap.** 64 KB per session, taken from both ends of the transcript — the
opening states intent, the ending states the outcome, and those are what people
search for. Artifacts and file paths are indexed first, since they are the
distilled signal.

## Migrations

`PRAGMA user_version` tracks the applied version. `migrations()` returns an
ordered list; each entry runs at most once, inside a transaction, and rolls
back on failure.

Opening a library written by a *newer* Artix is refused with an explicit
message rather than attempted and corrupted.

Adding a migration:

1. append `(2, include_str!("migrations/002_whatever.sql"))` to `migrations()`;
2. bump `SCHEMA_VERSION`;
3. if the change affects indexed text, bump the reindex path too — existing
   FTS rows are not rewritten automatically.

## Maintenance

`vacuum` (Settings → *Compact*) runs, in order:

```sql
PRAGMA wal_checkpoint(TRUNCATE);
PRAGMA optimize;
INSERT INTO session_fts(session_fts) VALUES('optimize');
VACUUM;
```

`VACUUM` cannot run inside a transaction or with statements pending, so it is
issued in its own connection scope.

## FTS5 availability

`rusqlite` is built with the `bundled` feature, which compiles SQLite from
source with FTS5 enabled — Artix never depends on whatever SQLite the host OS
happens to ship. Startup verifies this by probing
`pragma_compile_options` and, failing that, by attempting to create a temporary
FTS5 table. If it is genuinely missing, Artix refuses to open the library with
an actionable message instead of failing later at the first search.
